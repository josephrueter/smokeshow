# AGENTS.md

Two people work in this repo — Kelly (kvohs) and Joe (josephrueter) — each in
their own clone, both using coding agents. Nobody reviews anybody's work.
These rules plus automation are what keep us from stepping on each other.

## How work flows (generic — same in every repo we share)

- `main` is the live app. `staging` is the standing test environment.
  All work merges into `staging`. Only `staging` merges into `main`.
- Before starting anything: `git fetch origin` and branch off the latest
  `origin/staging`. Never work directly on `main` or `staging`.
- Branch names: `kelly/<short-description>` or `joe/<short-description>`.
  If your agent auto-created a branch under another prefix, rename it
  before opening the PR.
- Run agents in a git worktree so parallel work can't collide on disk.
- One task per branch. Keep changes scoped — no drive-by refactoring or
  reformatting of files the task doesn't touch.
- Small, focused commits with clear messages.
- Merge fast: open the PR into `staging`, enable auto-merge, and it merges
  itself when the build check goes green. Branches live hours, not days.
- Shipping to prod: when staging looks good, open a PR from `staging` to
  `main`. Either person may do this at any time, so anything merged to
  staging must be releasable.
- Before starting, glance at open PRs and branches. If the other person is
  working in an area, stay out of it or wait.
- Don't touch lockfiles, config, or CI files unless the task requires it,
  and say so in the PR description when you do.

## Hard rules — no exceptions without a human's OK

- Never push directly to `main` or `staging`.
- Never force-push a shared branch or rewrite pushed history.
- Never run destructive commands (`git reset --hard`, `git clean -fd`,
  deleting the other person's branches).
- Never merge with a failing or skipped build check.
- Never resolve a merge conflict by discarding the other side's changes.
  If a conflict isn't trivial, stop and ask a human.

## This repo: SMOKESHOW

Wildfire smoke forecast app. Product docs live in `docs/` — read
`smokeshow-build-brief.md` before product work. Stack today: React + Vite +
Leaflet on Vercel, edge functions in `api/`. If the stack changes, update
this section, not the rules above.

- Install `npm install` · run `npm run dev` · build `npm run build`
- No test suite; a passing build is the merge gate. Visual checks in
  `scripts/verify-*.mjs` need a running dev server and local Chrome — use
  them when touching map/scrubber/strip UI.
- API keys (`AIRNOW_API_KEY`, `PURPLEAIR_API_KEY`) are optional locally;
  without them sensor/history data is empty but the app runs.

Do not touch without understanding:

- The `data` branch — machine-owned; CI force-pushes smoke frames to it
  several times a day. Never commit to it or branch from it.
- `assets/smokeshow-*.svg` — generated by `assets/gen_smokeshow_art.py`.
  Edit the generator, never the SVGs.
- `api/` imports from `src/lib/` — a "frontend-only" change there can
  break share links and OG images.
- Cache headers in `api/aq.js` — they keep traffic spikes from blowing the
  Open-Meteo free tier.
- The `/s` → `/api/s` rewrite in `vercel.json` — powers every shared link.
- Disclaimer/health copy ships verbatim from the build brief. Past hours
  are "model estimate," never "observed." Don't reword without a human.
